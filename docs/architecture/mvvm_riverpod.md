# MVVM & Riverpod - Patterns & Conventions

Résumé : le projet suit MVVM : `Views` (UI) utilisent des `ViewModels` (StateNotifier ou Provider) qui orchestrent les `Services`.

Points clés :

- `ViewModels` exposent des providers (StateNotifierProvider, FutureProvider) définis dans `lib/viewmodel/`.
- Les `Views` (ou `screens`) consomment ces providers via `ref.watch` ou `ref.read`.
- Ne pas mettre de logique métier lourde dans les widgets : tout ce code va dans `viewmodel/` ou `services/`.

Exemples présents dans le code :

- `transfer_viewmodel.dart` (controle du `ShareEngine`).
- `selection_viewmodel.dart` (sélection d'items entre onglets).

Bonnes pratiques

- Providers courts et réutilisables.
- Eviter les side-effects dans `build()` — utiliser `WidgetsBinding.instance.addPostFrameCallback` pour actions post-render.
- Tests unitaires ciblant `ViewModels` plutôt que widgets côté logique.
